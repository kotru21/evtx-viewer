# evtxview — контекст для продолжения разработки

CLI для просмотра Windows `.evtx` на Linux. Написан в ходе DFIR-расследования ransomware-инцидента, где штатный `python-evtx` **молча читал только первый chunk из 30** (94 записи вместо 2309) и чуть не сорвал выводы. Поэтому инструмент построен на rust-based парсере `evtx` (корректно читает все chunk'и) и имеет встроенную **проверку полноты** (`--verify`), сверяющую число прочитанных записей с суммой по заголовкам chunk'ов.

## Установка

```bash
pip install -e .          # ставит evtxview + зависимость evtx, создаёт команду `evtxview`
# для разработки (тесты, линт):
pip install -e ".[dev]"
```

После установки доступны оба варианта запуска — команда `evtxview ...` и `python -m evtxview ...`. Работает одинаково на Windows и Linux (вывод принудительно в UTF-8, чтобы кириллица не билась на Windows-консоли).

Единственная runtime-зависимость — пакет `evtx` (rust-биндинг, класс `PyEvtxParser`). Всё остальное — стандартная библиотека Python.

## Что умеет сейчас

| Функция | Флаг | Описание |
|---|---|---|
| Проверка полноты | `--verify` | Сверяет прочитанное с заголовками chunk'ов **и** проверяет отсутствие пропусков в `EventRecordID`. Ловит молчаливую обрезку парсера, печатает `OK` или `!!! ОБРЕЗКА` со списком пропущенных ID. Пустые предвыделенные chunk'и (sentinel-заголовок) не считаются потерей |
| Сводка | `--summary` | Распределение EventID + диапазон времени. Security-relevant EID подсвечены |
| Фильтр по EID | `--eid 1,3,1102` | Через запятую |
| Фильтр по подстроке | `--grep spoolsv` | Регистронезависимый поиск в сыром XML |
| Фильтр по времени | `--after / --before "YYYY-MM-DD HH:MM"` | Время в UTC |
| Полный дамп | `--full` | Все поля события |
| Экспорт | `--csv FILE` / `--json FILE` | Метаполя (`_EventID`, `_UTC`, `_Local`, ...) + все Data-поля |
| Сдвиг TZ | `--tz 3.0` | Локальное время (по умолч. +3). Показывает и local, и UTC |
| Лимит | `--limit N` | Не более N событий |
| Несколько файлов | `evtxview *.evtx ...` | Маски раскрываются |

Обрабатывает битые chunk'и (не падает, считает и сообщает `chunk-errors`). Работает с `UserData`-форматом (PrintService и подобные), не только `EventData`. Цвета только при выводе в tty.

## Примеры (из реального кейса)

```bash
evtxview Sysmon.evtx --verify                    # chunks=30 заявлено=2309 прочитано=2309 OK
evtxview Sysmon.evtx --summary
evtxview Sysmon.evtx --eid 3 --grep spoolsv      # reverse shell из спулера
evtxview Sysmon.evtx --eid 1 --grep "net user"   # создание учёток
evtxview Sysmon.evtx --eid 1 --csv procs.csv
evtxview Security.evtx --eid 1102 --full         # очистка логов
evtxview *.evtx --verify                          # проверить все на обрезку
```

## Архитектура (для навигации по коду)

Весь код — в `src/evtxview/cli.py` (разбиение на модули запланировано в Фазе 1).

- `read_records(path)` — читает все записи через `PyEvtxParser`, ловит `RuntimeError` на битых chunk'ах
- `verify_completeness(path, seen_ids)` — ручной разбор заголовков chunk'ов (`ElfChnk\x00`, шаг 0x10000). Сверяет счётчик (номера записей, offset 0x08/0x10) **и** ищет пропуски в фактических `EventRecordID` против диапазона из заголовков (offset 0x18/0x20). Пустые chunk'и (sentinel `0xFFFF…FF`) игнорируются
- `get_record_id(xml)` — извлекает `EventRecordID` (для сверки полноты)
- `parse_record(xml) -> EventRecord` — **основной путь разбора**: парсит запись один раз через `ElementTree` (устойчив к namespace, атрибутам, многострочным значениям; декодирует XML-сущности `&lt;`→`<`). При битом XML — fallback на regex-функции ниже
- `EventRecord` — датакласс разобранной записи (`eid`, `record_id`, `utc`, `provider`, `computer`, `data`)
- `get_eid/get_utc/get_provider/get_computer/get_data_fields` — извлечение полей регулярками (fallback-путь для `parse_record` + сверка)
- `to_local(utc, offset)` — конвертация UTC→локальное
- `summarize_line` / `full_dump` — форматирование вывода
- `HOT_EID` — множество security-relevant EventID для подсветки
- `main()` — argparse, фильтрация, экспорт

## Известные ограничения / TODO (идеи для развития в Claude Code)

Осознанные упрощения текущей версии, которые стоит доработать:

1. ~~**Парсинг полей регулярками, а не XML-парсером.**~~ — **сделано.** `parse_record()` разбирает через `ElementTree` (namespace/атрибуты/многострочные значения, декодирование XML-сущностей), с fallback на regex для битого XML. На корпусе из 137 файлов: 0 потерянных значений, +396 полей, которые regex пропускал, декодированные `&lt;`→`<`.
2. **`--after/--before` только UTC.** Нет флага для указания, что фильтр в локальном времени. Стоит добавить `--tz-filter`.
3. **Нет пресетов.** Напрашиваются `--preset process-tree` (EID 1 с построением дерева parent→child по ProcessGuid), `--preset logon-analysis` (4624/4625/4634 + LogonType + Source IP), `--preset network` (EID 3 сгруппированный по dst).
4. **Нет корреляции между файлами.** Полезно было бы строить единый таймлайн из нескольких EVTX сразу, сортированный по времени (сейчас `*.evtx` обрабатывает файлы последовательно, не сливает).
5. **Sysmon-специфика захардкожена в HOT_EID и в выборе полей.** Стоит вынести маппинги EID→важные-поля в конфиг/словарь, чтобы легко расширять под другие провайдеры.
6. **Экспорт в таймлайн-форматы.** Прямой экспорт в формат IRIS/Timesketch/plaso был бы удобен (сейчас общий CSV).
7. **Нет обработки очень больших файлов потоково** — всё грузится в память списком. Для многогигабайтных логов стоит сделать генератор.
8. ~~**`verify_completeness` не учитывает hard-corrupted chunk'и**~~ — **сделано.** `--verify` теперь сверяет фактические `EventRecordID` с диапазоном из заголовков и печатает конкретные пропущенные ID; пустые предвыделенные chunk'и не считаются потерей.

## Происхождение / зачем такой дизайн

Ключевой урок кейса, зашитый в инструмент: **число из `records()` без ошибки ≠ полнота данных**. Парсер может тихо отдать часть. Поэтому `--verify` — не опциональная приблуда, а центральная функция: перед любым серьёзным анализом EVTX прогнать `--verify`, убедиться что прочитано всё. Это спасает от построения выводов на 5% лога.

Кросс-валидация приветствуется: на критичных артефактах сверять с `EvtxECmd` (Eric Zimmerman) или нативным `wevtutil` — расхождение сразу видно.
